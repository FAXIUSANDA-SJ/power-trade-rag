# 目标架构设计

## 1. 设计原则

- 单一生产后端：统一 Spring Boot 主链路。
- 存储与计算解耦：原始文件、元数据、向量索引、检索引擎各司其职。
- 摄取与问答解耦：文档入库改为异步任务化。
- 配置可治理：提示词、知识库配置、检索参数全部入库并版本化。
- 场景优先：针对电力交易的时效性、溯源性、版本性做专项设计。

## 2. 目标架构分层

### 2.1 接入层

- Web 前端：保留现有 Vue3 前端。
- API 层：统一由 Spring Boot 暴露 REST API。
- 认证鉴权：后续补充用户、角色、知识库权限边界。

### 2.2 业务层

- 会话问答域：负责会话、上下文、回答生成、引用返回。
- 知识库管理域：负责知识库、文档集、状态、版本、发布。
- 文档摄取域：负责上传、解析、分块、向量化、发布。
- 配置治理域：负责提示词配置、检索参数、知识库参数版本化。
- 运维评测域：负责任务监控、检索评测、质量分析。

### 2.3 基础能力层

- 对象存储：MinIO 存放原始文件、解析中间产物、OCR 结果。
- 关系库：MySQL 存放文档元数据、知识库、配置版本、任务状态。
- 缓存与队列：Redis 存放缓存、轻量任务队列、幂等控制。
- 向量库：Milvus 作为主向量库。
- 检索引擎：中期引入 OpenSearch，与 Milvus 组成混合检索。
- 模型层：统一模型网关，支持 Embedding、LLM、OCR/VLM。

## 3. 文档摄取目标链路

```text
上传文件
  -> 保存到 MinIO
  -> 写入 document_info / ingest_task
  -> 异步 Worker 拉取任务
  -> 文档解析器分流
  -> OCR / 表格识别 / 文本清洗
  -> Chunk 切分
  -> Embedding
  -> 写入 Milvus
  -> 写入检索索引
  -> 发布到知识库可检索状态
```

### 3.1 当前已落地的最小骨架

- 已补 `ingest_task` 任务表，用于记录文档摄取任务状态。
- 已补 `pending / running / success / failed` 基础状态流转。
- 当前上传接口已改为“先建任务再返回”，后台 Worker 通过轮询执行摄取任务。
- 已在上传返回中补充 `ingestTaskId` 和 `ingestStatus`，便于后续前端接入任务追踪。
- 上传接口当前已收口为“任务受理 DTO”，语义上明确表示“已接收、后台处理中”。
- 已补任务查询与重试接口，具备最小任务运营能力。
- 已补任务分页、状态筛选、按知识库筛选能力，便于后续运营和排障。
- Worker 已补原子抢占逻辑，降低多实例重复消费同一任务的风险。
- Worker 已补“已存在向量直接成功”的幂等保护，减少重复向量化。

## 4. OCR 与文档解析设计

## 4.1 为什么补 OCR

- 电力交易场景大量存在扫描件、公告 PDF、图片型附件、表格型规则。
- 当前仅依赖 Tika，无法稳定识别扫描文档、版面结构、复杂表格。
- 不补 OCR，后续即使向量库升级，召回质量也会被错误解析拖垮。

## 4.2 参考 RAGFlow 的设计思路

- 按文件类型分流解析器，而不是单一解析器通吃。
- 对 PDF、图片、扫描件增加 OCR 和版面识别。
- 对表格文档增加结构化抽取，不只抽纯文本。
- 解析后保留结构化 metadata，例如页码、标题层级、表格位置。

## 4.3 推荐解析分流

- PDF 文本型：优先文本解析器。
- PDF 扫描型：走 OCR + 版面识别。
- 图片型：走 OCR / VLM。
- Word / Markdown / TXT：走轻量文本解析。
- Excel / 表格型附件：走表格解析器，保留行列结构。

## 5. 向量库与检索演进

## 5.1 近期方案

- 以 Milvus 作为主向量库。
- 检索仍由应用层先完成向量召回主链路。
- 先满足稳定入库、删除、过滤、按知识库隔离。

## 5.2 中期方案

- 演进为 Milvus + OpenSearch。
- Milvus 负责语义向量召回。
- OpenSearch 负责关键词、BM25、过滤、聚合。
- 应用层负责召回融合、Rerank、去重和引用控制。

## 5.3 为什么不是一开始就全量上 OpenSearch

- 当前系统基础链路尚未统一，先上双引擎会放大复杂度。
- 先把文档解析、对象存储、向量主链路跑稳，收益更直接。
- 当规则类问答、关键词查准率成为瓶颈时，再引入 OpenSearch 成本更合理。

## 6. 配置入库并版本化

## 6.1 范围

- 提示词配置：`assistantName`、`systemPrompt`、`fallbackReply`、`memoryRounds`。
- 知识库配置：向量模型、解析策略、分块策略、是否启用 OCR、知识过滤规则。
- 检索参数：`topK`、`minScore`、召回模式、Rerank、引用上限、混合检索权重。

### 6.1.1 当前已落地内容

- 提示词配置：已接入统一版本表，支持版本列表和激活切换。
- 检索参数：已接入统一版本表，`RagCoreService` 已改为动态读取活动版本。
- 知识库配置：已接入统一版本表，创建知识库时自动生成默认配置版本。

## 6.2 为什么要这么做

- 本地文件方式不适合多实例部署，配置会漂移。
- 硬编码参数无法回溯，线上问题难以排查。
- 没有版本号就无法比较“这次效果变差到底改了什么”。
- 没有发布/回滚机制，线上调参风险大。
- 没有历史留痕，无法做审计和经验沉淀。

## 6.3 这么做的好处

- 可追踪：知道每次配置是谁、何时、因为什么修改。
- 可回滚：参数调坏了可以快速恢复上一个稳定版本。
- 可灰度：可先让某个知识库或环境试用新配置。
- 可评测：同一批问题可对比不同配置版本效果。
- 可多租户：后续不同知识库、不同业务线可用不同配置。

## 6.4 执行步骤

### 第一步：收口配置来源

- 去掉本地 JSON 作为生产真源。
- 去掉核心检索参数硬编码。
- 保留 YAML 仅作为基础设施和默认兜底配置。

### 第二步：设计配置表

- `prompt_config_version`
- `knowledge_base_config_version`
- `retrieval_config_version`
- `config_publish_record`

### 第三步：增加版本状态

- 草稿
- 已发布
- 已回滚
- 已废弃

### 第四步：增加读取规则

- 按环境读取生效版本。
- 按知识库读取专属版本。
- 若无专属版本，回退到系统默认版本。

### 第五步：增加发布与回滚接口

- 新建版本
- 查看版本列表
- 发布版本
- 回滚到指定版本
- 查看版本差异

## 6.5 操作难度评估

- 提示词配置版本化：中等偏低，改造点明确。
- 检索参数版本化：中等，涉及读取链路收口。
- 知识库配置版本化：中等偏高，涉及知识库模型、任务链路和前端交互。
- 整体难度：中等，不难在“建表”，难在“统一读取路径和发布流程”。

## 7. 电力交易场景专项设计

- 文档 metadata 增加：省区、市场类型、发文机构、发布日期、生效日期、失效日期、文档类型、业务主题。
- 回答默认返回：来源文档、片段、版本、生效状态。
- 规则类问题优先支持直接引用模式。
- 数据类问题后续通过结构化查询能力接入，不混进纯文档 RAG。

## 8. MinIO 接入说明

- 当前已完成 MinIO SDK 接入，并在服务启动时自动检查 bucket 是否存在。
- 文档上传在启用 `rag.minio.enabled=true` 时会直接写入 MinIO。
- 当前对象命名规则为 `docs/{uuid}{extension}`。
- 当前返回的存储路径形如 `minio://bucket/objectKey`，后续可继续扩展为预签名下载地址。

## 9. OCR Provider 预留

- 当前 OCR 能力已抽象为 `OcrProvider` 机制，支持按配置选择 provider。
- 默认 provider 为 `none`，不会触发真实 OCR。
- 已预留 `http` provider 接入位，后续可替换为内部 OCR 服务或第三方 OCR 网关。
- `http` provider 当前支持通用 JSON POST 方式，可配置 `endpoint/apiKey/responseField`。
- 已补 OCR 联调测试接口，便于在不改主链路的前提下单独验证 OCR 服务连通性与返回格式。

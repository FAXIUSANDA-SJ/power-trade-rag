# 架构整改任务清单

## 1. 目标与边界

- 当前阶段目标：先完成架构设计、技术方案、任务拆分和文档沉淀。
- 当前阶段不做：业务侧功能逻辑变更，只整理业务改动清单，待架构验收后再开始。
- 实施原则：按“统一主线、先收口再扩展、先文档后实现、先基础设施后业务能力”推进。

## 2. 总体执行策略

- 主线后端统一为 Spring Boot，Node `server.js` 退出生产链路。
- 先做模块化单体整改，再按需要演进到异步任务化和服务化。
- 文档处理改为“上传 -> 落存储 -> 入任务 -> 异步解析 -> 向量化 -> 发布生效”。
- 配置中心改为“入库 + 版本化 + 发布/回滚”，禁止继续以本地文件作为生产配置真源。

## 3. 优先级排序

### P0：必须先处理

#### T1. 统一后端主链路
- 问题：Java 与 Node 双后端并存，接口、端口、契约和数据链路不一致。
- 处理方案：Spring Boot 作为唯一生产后端；Node 服务仅保留为本地 mock 或下线。
- 输出物：后端主线说明、接口归一清单、淘汰路径。
- 依赖：无。
- 是否可并行：否，属于总控任务。

#### T2. 收口模型/向量库配置
- 问题：Qwen/OpenAI、内存向量库/Chroma/Milvus 多套配置并存，存在 Bean 冲突风险。
- 处理方案：统一配置来源；保留单一生效路径；按 profile 切换环境；移除重复 Bean。
- 输出物：配置矩阵、配置规范、冲突清理清单。
- 依赖：T1。
- 是否可并行：可与 T3 文档设计并行，不可与最终实现完全并行。

#### T3. 输出目标架构设计
- 问题：当前只有“现状代码”，缺少正式目标架构和演进路线。
- 处理方案：补齐目标架构、部署架构、文档摄取链路、检索链路、版本化配置方案。
- 输出物：架构设计文档、README 文档入口、后续优化路线。
- 依赖：无。
- 是否可并行：是。

### P1：架构闭环核心任务

#### T4. 引入对象存储 MinIO
- 问题：当前文档落本地磁盘，不适合容器化、多实例、生产运维。
- 处理方案：文件上传后统一存入 MinIO；数据库仅保存对象 Key、版本、元数据。
- 输出物：存储设计、对象命名规范、迁移方案。
- 依赖：T3。
- 是否可并行：是。

#### T5. 文档摄取链路异步化
- 问题：上传接口同步解析和向量化，失败恢复与状态追踪能力弱。
- 处理方案：增加任务表/任务状态机；改为异步处理；支持失败重试。
- 输出物：任务状态设计、处理时序图、表结构方案。
- 依赖：T1、T2、T4。
- 是否可并行：部分可并行。

#### T6. 接入 OCR / 深度文档解析能力
- 问题：当前主要依赖 Tika，无法覆盖扫描件、复杂表格、版面型 PDF。
- 处理方案：参考 RAGFlow 设计解析分流；增加 OCR、版面识别、表格提取能力。
- 输出物：解析器分流方案、OCR 技术选型、接入流程。
- 依赖：T5。
- 是否可并行：可与 T7 文档方案并行。

#### T7. 向量检索演进到 Milvus，预留 Milvus + OpenSearch
- 问题：当前只有单路向量检索，缺少全文与混合检索能力。
- 处理方案：近期主选 Milvus；中期演进为 Milvus + OpenSearch，实现混合检索。
- 输出物：技术选型对比、部署方案、演进路线。
- 依赖：T2、T3。
- 是否可并行：是。

### P2：质量和可运营能力

#### T8. 提示词配置、知识库配置、检索参数入库并版本化
- 问题：配置分散在文件、YAML、硬编码中，不可审计、不可回滚、不可灰度。
- 处理方案：统一建表，按版本管理，支持草稿/发布/回滚/生效历史。
- 输出物：表设计、接口设计、页面交互设计、发布流程。
- 依赖：T3。
- 是否可并行：是。

#### T9. 检索能力升级
- 问题：当前仅有向量召回，没有混合召回、Rerank、元数据过滤治理。
- 处理方案：引入向量召回 + 关键词召回 + Rerank + 引用控制。
- 输出物：检索参数设计、召回融合策略、验收指标。
- 依赖：T7、T8。
- 是否可并行：部分可并行。

#### T10. 引用溯源与知识版本治理
- 问题：回答缺少来源、版本和生效时间，无法满足电力交易场景要求。
- 处理方案：回答返回来源文档、片段位置、知识版本、生效时间。
- 输出物：引用返回结构、知识版本字段设计。
- 依赖：T8、T9。
- 是否可并行：部分可并行。

### P3：后续增强

#### T11. 结构化数据与工具调用接入
- 处理方案：把交易价格、成交量、规则参数等数据能力通过 SQL/API 工具接入。
- 备注：属于业务增强，放到架构验收后。

#### T12. 评测、监控、运营闭环
- 处理方案：补齐评测集、检索命中率、回答引用率、任务成功率、模型成本监控。
- 备注：建议在基础链路稳定后接入。
